# Web 控制台使用指南

## 启动

在项目根目录运行：

```bash
python ui/server.py --host 127.0.0.1 --port 8765
```

浏览器打开：

```text
http://127.0.0.1:8765/
```

如果端口已被占用，可以换一个端口：

```bash
python ui/server.py --host 127.0.0.1 --port 8766
```

## API key

真实模型 API key 仍然通过环境变量配置。页面只显示 key 是否就绪，不显示 key 内容。

PowerShell 示例：

```powershell
$env:DEEPSEEK_API_KEY="your_deepseek_api_key"
$env:GLM_API_KEY="your_glm_api_key"
python ui/server.py --host 127.0.0.1 --port 8765
```

## 页面能力

- 选择一个或多个模型。
- 选择测试类别：`audit` 或 `log_analysis`。
- 按题库来源过滤任务。
- 勾选具体任务；如果不勾选，则运行当前类别和来源下的全部匹配任务。
- 设置测试次数、最大任务数、最大输出 token 和 run id。
- 执行运行前自检。
- 预览运行计划。
- 启动评测并查看执行日志、完成数量、百分比、耗时和当前步骤。
- 自动生成 HTML/Markdown 报告。
- 在历史报告列表中打开报告和图表。

## 进度说明

运行进度按“模型 × 任务 × 测试次数”估算总量。每完成一个模型-任务-轮次组合，进度会前进一步。

页面会展示：

- 已完成 / 总数。
- 百分比。
- 当前耗时。
- 当前步骤，例如正在执行评测、正在生成报告、评测完成。

如果某个模型 API 调用很慢，页面不会卡死，执行日志和耗时仍会持续刷新。

## 注意

Web 控制台复用命令行 runner，不绕过原有安全边界、评分器和报告生成器。当前版本是本地单用户控制台，不建议暴露到公网或多人共享网络。

当前 UI 风格参考了开源后台控制台的常见布局思路，例如顶部状态摘要、左侧配置表单、右侧任务表和运行日志。实现仍是本地自包含 HTML/CSS/JS，没有复制第三方模板文件。

现阶段暂不引入 React/Vite 等构建链，原因是平台还在验证评测主流程，本地零依赖启动更重要。后续如果要做用户管理、复杂筛选、报告内嵌预览和多页面路由，可以再切换到成熟前端栈。

模型展示会同时包含平台内的模型名称和厂商/配置里的具体型号，例如：

```text
deepseek / deepseek-v4-pro
glm / glm-5.2
```

## v0.20 类别说明

当前 UI 会从后端任务接口读取可用测试类别。已接入的主线类别包括：

- `audit`：代码审计，判断漏洞类型、位置、根因、利用方式和修复建议。
- `log_analysis`：日志/告警分析，判断攻击类型、证据、时间线、影响和处置建议。
- `patch`：安全修复，评估模型是否能提出最小、可落地、低副作用的修复方案和测试用例。
- `ctf`：CTF 解题，评估模型是否能从附件或题面中还原 flag，并给出关键证据和复现步骤。
- `tool_use`：受控工具使用计划，评估模型是否能选择正确工具、设置参数、安排步骤、形成证据链并遵守安全边界。

UI 是测试平台的控制台入口；主线能力仍然是最初方案中的五类测试目标。

## 最大输出 token

“最大输出 token”留空时，平台使用当前测试类别的默认值。当前 `audit`、`log_analysis`、`patch`、`ctf`、`tool_use` 默认都是 `4096`。

如果需要临时提高或降低某次测试的输出上限，可以在 UI 中填写具体数字；这会覆盖类别默认值，并写入结果中的 `max_tokens_source=override`。
